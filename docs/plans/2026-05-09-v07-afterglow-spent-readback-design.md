# v0.7 余辉已投入读回设计

## 背景

v0.7 `归航航线` 已能说明当前段收益、下一段目标、差距、行动提示和下一段预览。手动复核发现一个读回断点：`归航余辉` 只在当前星尘仍高于余辉起步值时显示。一旦玩家把起步星尘花在自动采集器上，`共鸣矩阵` 会退回“共鸣暂存”，削弱“余辉已经帮我重建”的反馈。

## 方案

保留 `归航余辉` 公式、50 星尘上限、存档字段和 `归航航线` 门槛不变。UI 只调整派生读回：

- 有额外共鸣且已满 2 个永久节点时，持续显示 `归航余辉` 结构化读回。
- 当前星尘仍不少于余辉起步值时，沿用“可立即重建 N 台自动采集器”文案。
- 当前星尘已低于余辉起步值时，显示“本轮起步获得过 X 星尘，若已花掉则表示余辉已投入重建节奏”。

## 验证

- 先新增 SSR 测试，构造 `dust: 0`、额外共鸣和两永久节点已满的状态，要求仍显示余辉已投入文案，并不再显示“等待后续版本扩展用途”。
- 最小实现只改 `src/App.tsx` 的展示条件和文案分支。
- 标准验证链路：`bun test`、`bun run test`、`bun run build`、`./ops/governor-check.sh`、`git diff --check`。

## 非目标

- 不新增资源、节点、节点等级、任务系统、多生产线、额外面板、存档字段、奖励公式、余辉公式、视觉资产或 telemetry。
