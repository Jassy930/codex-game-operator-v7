# v0.7 Return Route Design

## Problem

v0.6 `归航余辉` 已让满节点后的重复归航产生起步价值，但 20 小时后目标仍容易退化成“继续归航，等后续版本”。下一步需要一个能承接多次归航的目标结构，同时保持首分钟安静、面板数量不变、系统数量不膨胀。

## Recommended Approach

v0.7 使用 `归航航线`：在现有 `共鸣矩阵` 和阶段目标里显示最多 3 个长期航线里程碑。航线只解释“下一次归航正在推进哪段长期目标”，不新增资源、不新增共鸣节点、不增加节点等级，也不引入任务系统。

## Alternatives Considered

- 提高 `归航余辉` 上限：实现简单，但只能让新一轮更快，不会形成 20 小时后的方向感。
- 新增第三共鸣节点：能制造选择，但会扩大矩阵复杂度，并绕开当前“最多 2 个永久节点”的边界。
- 节点等级树：长期空间最大，但正是当前预算明确禁止的复杂度来源。

## Budget

采用 `docs/COMPLEXITY_BUDGET.md` 的 `v0.7 Return Route Budget`：

- 最多 3 个航线里程碑。
- 复用 `共鸣矩阵` 和阶段目标。
- 不新增存档字段，优先从 `returnCount`、已启动永久节点和可用共鸣推导读回。
- 不新增第三普通资源、第三共鸣门槛、新节点、节点等级、节点等级树、任务系统、多生产线、额外面板或 telemetry。

## Next Implementation Slice

下一轮应先做一个可玩读回切片：当玩家已有归航历史并进入满节点后的重复归航阶段，`共鸣矩阵` 显示当前航线段、下一段门槛和为什么继续归航有意义。实现前必须用 TDD 覆盖航线段推导和 UI 读回。
