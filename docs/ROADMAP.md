## 当前阶段闸门：v0.7 / 归航航线

进入下一轮玩法扩展前，operator 应优先检查：

- 前 60 秒是否仍不被 `共鸣矩阵` 干扰。
- `星尘引擎室` 后玩家是否能理解共鸣门槛：20 台自动采集器和 12 次调校。
- 首次领取共鸣后，三选一永久节点是否形成清楚选择。
- 两个永久节点已满后，`归航余辉` 是否让下一轮起步更可感知，并能读懂“先重建前 3 台自动采集器”。
- 20 小时后是否能读懂当前航线段、带量化收益的航线摘要、当前收益、余辉已投入本轮重建、下一段归航目标、距下一段还差多少、下一步行动提示、节奏预判、航线图、下一段预览，以及继续归航的意义。
- 是否存在真实反馈或 self-playtest gap 支撑下一步，而不是直接追加节点等级树、第三节点或任务系统。

允许的下一步仍必须是小切片：既有航线读回调整、self-playtest 复核、恢复远端反馈读取或等待真实反馈。当前 metrics snapshot readback 已能返回 `activeSessionDurationMs`；后续不要继续扩 metrics 字段，除非有新的可解释运营缺口。禁止把 v0.7 直接扩成任务系统、多生产线、更多节点、第三个共鸣里程碑、节点等级树或多个新面板。

## 阶段推进节奏

不要在同一阶段停留太久。

- 同一时间窗连续 2 轮 no-change 后，必须做 stage review。
- stage review 只能得出三类结论：扩大时间窗、定义内容弧线、或明确等待真实反馈。
- 如果扩大时间窗，下一轮检查必须写清新的目标范围，例如 `15-60 分钟` 或 `首次回访`。
- 如果定义内容弧线，先写文档，不直接实现系统。
- 如果等待真实反馈，不继续产出重复 no-change 文档；转向反馈获取、metrics readback 或暂停新增玩法。

当前 `docs/CONTENT_ARC.md` 已覆盖 0-60 分钟、首次回访和 v0.3 回访后策略层。后续实现必须从该内容弧线、真实反馈、self-playtest gap 或 local-only metrics 出发。

最近阶段切换：用户明确要求更丰富玩法后，v0.3 已实现为 `共鸣系统` 第一版。用户进一步提出 20 小时目标后，v0.4 第一刀复用现有共鸣矩阵增加第二共鸣门槛和第二节点选择，并补齐第二共鸣未消费目标优先级、双节点状态下的回访计划读回、第二节点启动效果反馈、`2/2` 选择上限读回、20 小时巡航读回、双节点组合读回和当前版本共鸣完成读回。本地指标快照读回入口已补齐，并可读回活跃 session 当前持续时长。当前用户方向已经明确转向 prestige 长线主机制，v0.5 已完成 `星尘归航` 预算、存档迁移、归航逻辑、现有矩阵 UI 第一版、归航后第 2 轮开局共鸣读回和满节点额外共鸣暂存说明；v0.6 已把额外共鸣转成 capped `归航余辉` 起步星尘；v0.7 已规划并落地为现有矩阵内的 `归航航线`，现在补齐动态差距、行动提示、下一段预览、当前收益读回和带当前价值短语的摘要。星图巡航暂缓为后续辅助，不抢占归航主线。
